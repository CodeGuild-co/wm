Software engineering, in my opinion, boils down to two things:

Solving problems and being lazy

If you can do these things, you can engineer software

Problem Solving

The art of breaking a task down into small, single-focus problems, then solving those problems whilst being aware of the whole.

Let's take an example


Write a script that calculates the
nth number in the Fibonacci sequence

e.g.
> fibonacci(6)
8
> fibonacci(10)
55

What are the problems here?

Here's my list:

  • What's the Fibonacci sequence?
  • How do I calculate numbers in the sequence?
  • Who's asking?
  • What's a script?
  • How can we write one?
  • What platform are we on?
  • What language do we pick?
  • Who's going to use the script?
  • What do I need to install?
  • What do I need to deliver?
  • How fast does this need to be?
  • How far into the sequence should I be prepared to go?

Problems are often "meta problems", problems you have to solve in order to solve other problems

e.g. What's the Fibonacci sequence?

Some problems are specific to you the individual

e.g. What text editor should I use?

Some are shared problems between developers

e.g. What language should we use?

Some problems are specific to the project

e.g. How do I calculate numbers in the sequence?

I think this kind of method of problem solving; breaking things down into smaller chunks, is quite natural. I think most people do it all the time, how could you get any suitably large task done otherwise?

In software engineering you tend to need to be more explicit about what these tasks are, you need to write them down and describe them.

This is because you want to be able to stop working on the project and come back to it tomorrow.

You want to be able to share the project workload with other people.

You want to be able to easily tell what you're doing, why you're doing it, and in what context you're doing it.

You want to know how well you're doing, and how much more work is required before you finish.

That's problem solving...

Being Lazy

The second part of software engineering is being lazy

This means doing the smallest amount of work possible in order to solve the problem

  • Don't write code if someone else has already written it
  • Don't spend time making the code better and faster if you don't have to
  • DRY: Don't repeat yourself
  • KISS: Keep It Simple Stupid

Being lazy doesn't mean doing work badly

Don't cut corners, don't rush stuff, don't leave work for other people to do

It means, how can you most easily do an excellent job

Being lazy doesn't mean shooting yourself in the foot

There are lots of problems to solve for any given task, the laziest solution to one problem might mean lots of work further down the line

In fact being lazy might involve putting in hours of work now in order to save weeks of work later

That's it!
